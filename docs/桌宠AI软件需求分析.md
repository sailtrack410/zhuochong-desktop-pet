# 桌宠 AI 软件需求分析

## 1. 文档说明

本文档用于定义桌宠 AI 项目的正式需求基线，作为后续产品设计、架构设计、开发实施和测试验收的依据。

本文档目标：

- 明确项目目标和边界
- 统一功能范围和优先级
- 约束首版实现方向
- 为后续模块设计提供依据

本文档适用阶段：

- 原型阶段
- MVP 阶段
- 首版功能细化阶段

## 2. 项目概述

### 2.1 项目名称

桌面桌宠 AI

### 2.2 项目目标

开发一个运行于桌面的 AI 桌宠应用，使其具备：

- 持续陪伴感
- 轻量聊天能力
- 适度主动提醒能力
- 基础桌面助手能力
- 逐步形成对用户的长期记忆

项目核心不是单纯聊天，而是让用户愿意长期把它挂在桌面上。

### 2.3 目标优先级

功能目标优先级按如下顺序定义：

1. 陪伴感
2. 聊天
3. 提醒效率
4. 桌面助手

### 2.4 产品定位

产品定位为：

- 非商用、开源的桌面桌宠 AI
- 首先满足开发者本人日常使用
- 后续允许他人下载和自定义配置使用

## 3. 用户与使用对象

### 3.1 目标用户

首版主要目标用户为：

- 开发者本人

后续潜在用户为：

- 愿意尝试桌宠 AI 的个人用户
- 愿意自行配置模型接口的开源用户

### 3.2 用户特点

目标用户默认具备以下特征：

- 能接受本地桌面软件安装和配置
- 能自行填写模型 API 配置
- 期望桌宠有陪伴感，而不仅是工具
- 能接受一定程度的个性化与实验性功能

## 4. 使用场景

### 4.1 核心场景

- 用户工作或日常使用电脑时，桌面上持续显示一只小猫桌宠
- 用户点击小猫后弹出一个小聊天框，与桌宠进行文字互动
- 桌宠在适当场景下主动提醒用户休息、关注时间、查看日程
- 桌宠逐步积累关于用户的记忆，并在软件中以“记忆/日记”形式展示

### 4.2 典型使用流程

#### 场景 A：日常陪伴

1. 用户打开电脑，桌宠常驻显示
2. 小猫默认仅显示本体，不展开面板
3. 用户偶尔点击桌宠，查看聊天框或收到互动反馈
4. 用户在不繁忙时与其聊天

#### 场景 B：轻量提醒

1. 桌宠根据时间、电量、天气、系统空闲状态或日历信息做出提醒判断
2. 在允许打扰的情况下，以较轻量的方式弹出提示
3. 用户可忽略，也可点击进入聊天框继续互动

#### 场景 C：长期记忆

1. 用户日常聊天和互动
2. 桌宠自动提取认为重要的信息
3. 每天沉淀为记忆或日记内容
4. 用户在软件内查看记忆内容，但不在应用内修改

## 5. 已知条件

- 项目首版需要同时支持 Windows 与 macOS
- 项目时间目标为 1 周原型
- 首版功能要求真可用，不接受大量占位功能
- 默认桌宠形象为当前已有的像素猫
- API 由用户自行配置
- 模型接口应兼容 OpenAI 风格 API
- 聊天气泡和基础面板可先使用代码绘制
- 已有一款像素中文字体可用

## 6. 范围定义

### 6.1 首版必须实现

- 桌面常驻显示
- 拖拽移动
- 点击互动
- 文字聊天
- 主动提醒
- 自动记忆提取
- 记忆/日记查看
- 系统感知
- 设置页
- 托盘入口

### 6.2 首版明确不做

- 语音播报
- 复杂实时语音输入
- 高风险系统自动操作
- 将前台软件名称识别作为关键能力
- 商业化相关能力

### 6.3 平台差异允许项

项目首版要求同时支持 Windows 和 macOS，但允许系统能力存在差异。

已确认差异：

- macOS 可读取系统日历
- Windows 首版可不读取系统日历

## 7. 产品交互定义

### 7.1 默认桌面形态

默认状态下，桌面仅显示小猫本体。

要求：

- 常驻可见
- 不默认展开复杂界面
- 保持轻量和低打扰

### 7.2 主交互入口

用户通过点击小猫弹出小聊天框。

聊天框要求：

- 轻量
- 不遮挡主要工作区域
- 能承载基础对话和提醒交互

### 7.3 托盘交互

应提供托盘入口，用于：

- 打开设置
- 临时隐藏桌宠
- 退出软件
- 进入记忆/日记查看入口

## 8. 功能需求

## 8.1 桌宠显示模块

### 功能目标

在桌面上持续展示桌宠角色，并保持自然存在感。

### 功能要求

- 桌宠默认显示为小猫本体
- 支持桌面常驻
- 支持拖拽位置调整
- 支持点击触发互动
- 支持基础状态切换

### 输入

- 用户点击
- 用户拖拽
- 系统事件
- 内部状态切换

### 输出

- 小猫图像状态变化
- 面板弹出
- 轻量互动反馈

## 8.2 聊天模块

### 功能目标

允许用户与桌宠进行轻量、自然的文字交流。

### 功能要求

- 点击小猫后可打开小聊天框
- 用户可输入文字
- 桌宠返回文字回复
- 回复风格应符合陪伴型桌宠角色定位
- 支持基于上下文的连续对话

### 输入

- 用户文字消息
- 当前会话上下文
- 用户记忆摘要
- 系统环境信息

### 输出

- 桌宠文字回复
- 必要时触发记忆提取候选
- 必要时更新角色状态

## 8.3 主动提醒模块

### 功能目标

在不打扰用户的前提下，提供适度、有价值的主动提醒。

### 功能要求

- 主动性强度应支持用户配置
- 默认档位为“中等主动”
- 能根据时间、电量、天气、系统空闲状态、日历等信息判断是否提醒
- 提醒方式应轻量，不应过度干扰用户

### 默认静默规则

- 全屏时默认闭嘴
- 专注模式或免打扰时默认闭嘴

### 输入

- 时间
- 电量
- 天气
- 系统空闲/活跃状态
- 日历/提醒事项
- 用户主动性设置

### 输出

- 提醒文本
- 轻量气泡提示
- 相关状态变化

## 8.4 记忆与日记模块

### 功能目标

让桌宠逐渐认识用户，并形成可持续积累的长期记忆。

### 功能要求

- 记忆以自动提取为主
- 默认策略较开放
- 每天沉淀为“记忆/日记”
- 软件内提供查看能力
- 软件内默认只读
- 本地文件可由用户自行修改

### 记忆内容方向

- 用户偏好
- 用户习惯
- 用户近期重要事件
- 对用户关系有帮助的信息

### 不等同于

- 纯聊天记录
- 纯日志堆积

### 输入

- 聊天内容
- 用户使用过程中的重要信息
- 系统上下文

### 输出

- 结构化记忆
- 每日日记内容
- 查看页面可展示的数据

## 8.5 系统感知模块

### 功能目标

为桌宠提供适度环境感知，使主动行为和提醒更合理。

### 首版感知范围

- 时间
- 电量
- 天气
- 系统空闲/活跃状态
- 日历/提醒事项

### 说明

- 前台软件名称识别因跨平台稳定性问题，不作为首版关键能力
- 日历读取允许平台差异实现

### 输入

- 操作系统环境信息
- 第三方服务数据

### 输出

- 标准化环境上下文
- 提醒触发依据
- 对话上下文补充信息

## 8.6 设置模块

### 功能目标

允许用户完成运行所需的关键配置，并调整桌宠行为。

### 首版应包含

- 模型 API 配置
- OpenAI 兼容接口地址配置
- 模型名称配置
- 主动性强度配置
- 桌宠基础行为设置
- 可能的平台权限说明

### 输入

- 用户手动填写配置

### 输出

- 本地配置文件
- 运行时配置生效

## 8.7 托盘模块

### 功能目标

为桌宠提供轻量级的常驻入口和控制方式。

### 首版应包含

- 显示/隐藏桌宠
- 打开设置
- 查看记忆/日记入口
- 退出应用

## 9. 视觉与素材需求

### 9.1 角色素材

- 默认角色使用当前像素猫资源
- 角色是首版默认版角色
- 首版不强求复杂动画系统

### 9.2 UI 风格

- 允许基础 UI 采用代码绘制
- 聊天气泡和小面板首版无需复杂插画
- 应尽量保持与像素猫相协调的风格

### 9.3 仍需补充的素材

- 应用图标
- 托盘图标
- 必要的状态图标

## 10. 模型与接口需求

### 10.1 模型接入要求

- 支持 OpenAI 风格 API
- 不绑定特定平台供应商
- 用户自行填写 API Key 与地址

### 10.2 接口要求

- 至少支持文本对话能力
- 至少支持生成较自然的陪伴型回复
- 至少支持提取记忆候选或生成日记内容

## 11. 数据需求

系统需要至少保存以下数据类型：

- 用户配置
- 聊天记录
- 记忆数据
- 日记数据
- 主动提醒相关状态
- 平台能力差异配置

## 12. 非功能需求

## 12.1 平台需求

- 首版必须支持 Windows
- 首版必须支持 macOS

## 12.2 稳定性需求

- 应具备基本常驻稳定性
- 不应频繁卡死或崩溃
- 基本交互应可重复使用

## 12.3 可配置性需求

- 用户应可配置模型接口
- 用户应可调节主动性强度
- 用户应可在本地文件层面修改数据

## 12.4 开放性需求

- 项目应适合开源发布
- 不依赖无法替代的私有平台能力

## 12.5 体验要求

- 默认只显示小猫，不要过度打扰
- 点击交互应简单直观
- 聊天框应轻量
- 提醒应克制

## 13. 成功标准

1 周原型的成功标准定义如下：

- 桌宠可在桌面常驻显示
- 用户可拖拽、点击并打开小聊天框
- 用户可与桌宠进行稳定文字聊天
- 桌宠能基于部分环境信息进行主动提醒
- 桌宠能形成每天的记忆/日记沉淀
- 你本人愿意长期将其挂在桌面上使用

## 14. 主要风险

### 14.1 范围风险

- 双平台同时支持
- 时间仅 1 周
- 功能要求真可用

这三项叠加后，首版范围存在明显过大风险。

### 14.2 记忆风险

- 自动提取过于开放，可能导致记忆过多
- 可能产生记错、误判、低价值记录

### 14.3 平台差异风险

- Windows 与 macOS 系统能力接入差异较大
- 日历、电量、系统状态读取方式不同

### 14.4 体验风险

- 如果提醒过于频繁，会破坏陪伴感
- 如果聊天回复机械，会影响长期常驻意愿

## 15. 约束与边界

### 已确认约束

- 不做商用版本需求
- 不优先考虑语音交互
- 不将前台应用识别作为首版核心
- 软件内记忆只读
- 用户可直接修改本地文件

### 当前边界结论

首版必须首先证明：

- 这个桌宠在桌面上“待得住”
- 它聊起来不违和
- 它提醒得不烦
- 它能逐步记住用户

## 16. 后续必须继续细化的需求

虽然本文档已可作为总需求基线，但以下内容后续仍需单独细化成专题文档：

1. 小聊天框的字段与布局定义
2. 记忆/日记的数据结构定义
3. 自动记忆触发条件与写入规则
4. 主动提醒频率上限与策略规则
5. 设置页字段定义
6. Windows 与 macOS 平台能力差异清单
7. 图标与 UI 风格补充规范

## 17. 结论

本项目的首版定位已经明确：

它不是一个纯聊天客户端，也不是一个重度自动化助手，而是一个以陪伴感为核心、兼具聊天、提醒和长期记忆能力的桌面桌宠 AI。

后续开发应优先保证以下四点：

- 小猫始终自然地存在于桌面
- 点击互动轻量且顺手
- 主动提醒有价值且不过度打扰
- 记忆与日记机制能逐渐建立关系感

后续所有实现方案、模块设计和开发计划，应以本文档为需求基线，不随意偏离。
