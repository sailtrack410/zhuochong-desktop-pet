# 桌宠 AI 模块依赖与开发里程碑

## 1. 文档目的

本文档用于说明桌宠 AI 项目的模块依赖关系、开发先后顺序和推荐里程碑，帮助后续开发时保持主线稳定，避免模块之间相互阻塞。

本文档适用于自由开发节奏，不绑定固定工期。

## 2. 使用方式

后续开发时应遵循以下原则：

- 优先完成低依赖、高收益模块
- 尽量先做可独立验证的闭环
- 先打通“主流程”，再补“增强体验”
- 涉及平台差异的能力必须先抽象接口，再写平台实现

## 3. 一级模块列表

本项目当前一级模块如下：

1. 桌宠外壳模块
2. 基础交互模块
3. 聊天模块
4. 提醒与主动行为模块
5. 记忆与日记模块
6. 系统感知模块
7. 设置与配置模块
8. 托盘与应用控制模块
9. 数据与本地存储模块
10. 跨平台适配模块
11. UI 风格与资源模块
12. 调试与可观测性模块

## 4. 模块依赖总览

可以将依赖关系理解为四层：

### 第一层：基础底座

- 数据与本地存储模块
- 跨平台适配模块
- UI 风格与资源模块

### 第二层：可见壳体

- 桌宠外壳模块
- 基础交互模块
- 设置与配置模块
- 托盘与应用控制模块

### 第三层：业务能力

- 聊天模块
- 系统感知模块
- 提醒与主动行为模块
- 记忆与日记模块

### 第四层：辅助保障

- 调试与可观测性模块

说明：

- 调试模块虽然列在最后，但应尽早以最小版本接入
- 它不是业务功能，但会影响所有模块的开发效率

## 5. 详细依赖关系

## 5.1 数据与本地存储模块

### 被谁依赖

- 桌宠外壳模块
- 聊天模块
- 记忆与日记模块
- 设置与配置模块
- 提醒与主动行为模块

### 自身依赖

- 无

### 结论

这是底层基础模块，应尽早建立最小版本。

## 5.2 跨平台适配模块

### 被谁依赖

- 系统感知模块
- 桌宠外壳模块中的部分平台行为
- 托盘与应用控制模块中的平台差异逻辑

### 自身依赖

- 无

### 结论

这是双平台开发的底层基础，应在系统能力接入前先抽象。

## 5.3 UI 风格与资源模块

### 被谁依赖

- 桌宠外壳模块
- 基础交互模块
- 聊天模块的面板显示
- 托盘模块的图标资源

### 自身依赖

- 无

### 结论

默认像素猫资源和基础字体应尽早接入，否则前面几个可见模块都无法完整验证。

## 5.4 桌宠外壳模块

### 依赖

- UI 风格与资源模块
- 数据与本地存储模块

### 被谁依赖

- 基础交互模块
- 托盘与应用控制模块

### 结论

这是最先可见的主界面基础，应作为第一批开发对象。

## 5.5 基础交互模块

### 依赖

- 桌宠外壳模块
- UI 风格与资源模块

### 被谁依赖

- 聊天模块

### 结论

点击和拖拽是主入口，不应滞后于聊天模块。

## 5.6 设置与配置模块

### 依赖

- 数据与本地存储模块

### 被谁依赖

- 聊天模块
- 提醒与主动行为模块
- 系统感知模块的部分配置

### 结论

模型配置属于聊天模块的前置条件，因此设置模块必须较早完成最小可用版。

## 5.7 聊天模块

### 依赖

- 设置与配置模块
- 数据与本地存储模块
- 基础交互模块

### 被谁依赖

- 记忆与日记模块

### 结论

聊天是核心能力之一，但它不是第一步，必须在交互入口和配置基础具备后再接入。

## 5.8 系统感知模块

### 依赖

- 跨平台适配模块
- 设置与配置模块中的部分能力配置

### 被谁依赖

- 提醒与主动行为模块
- 聊天模块的环境上下文补充

### 结论

系统感知要先打通统一接口，再逐项扩展来源。

## 5.9 提醒与主动行为模块

### 依赖

- 系统感知模块
- 设置与配置模块
- 数据与本地存储模块

### 被谁依赖

- 无

### 结论

这是“陪伴感”和“效率感”的关键模块，但前置依赖较多，适合在聊天跑通后进入。

## 5.10 记忆与日记模块

### 依赖

- 聊天模块
- 数据与本地存储模块

### 被谁依赖

- 提醒与主动行为模块的增强逻辑

### 结论

记忆机制必须依托聊天或行为数据，不能脱离主流程单独开发。

## 5.11 托盘与应用控制模块

### 依赖

- 桌宠外壳模块
- 设置与配置模块
- UI 风格与资源模块

### 被谁依赖

- 无

### 结论

托盘属于重要入口，但不阻塞最初主流程，可以在外壳稳定后接入。

## 5.12 调试与可观测性模块

### 依赖

- 各模块最小日志入口

### 被谁依赖

- 所有开发阶段

### 结论

最少版本应在聊天接入前完成，否则后续问题定位成本会很高。

## 6. 推荐开发主线

推荐主线不是“按模块全部做完”，而是按闭环推进。

### 主线一：存在闭环

目标：

- 小猫能稳定显示在桌面
- 能拖拽
- 能点击

涉及模块：

- UI 风格与资源模块
- 数据与本地存储模块
- 桌宠外壳模块
- 基础交互模块

交付结果：

- 桌宠可见
- 位置可调整
- 状态图可切换

### 主线二：聊天闭环

目标：

- 点击小猫能弹出小聊天框
- 能配置模型接口
- 能完成真实对话

涉及模块：

- 设置与配置模块
- 聊天模块
- 调试与可观测性模块

交付结果：

- 用户可输入
- 模型可返回
- 历史消息可记录

### 主线三：提醒闭环

目标：

- 桌宠能在合适时机主动提醒

涉及模块：

- 跨平台适配模块
- 系统感知模块
- 提醒与主动行为模块

交付结果：

- 能读取环境信息
- 能执行基础提醒逻辑
- 能执行静默规则

### 主线四：记忆闭环

目标：

- 桌宠能记住用户并生成日记

涉及模块：

- 记忆与日记模块
- 数据与本地存储模块增强
- 聊天模块联动

交付结果：

- 自动提取记忆
- 生成每日记忆/日记
- 软件内可查看

### 主线五：桌面应用闭环

目标：

- 桌宠作为完整桌面应用更顺手

涉及模块：

- 托盘与应用控制模块
- 设置与配置模块增强
- UI 风格与资源模块增强

交付结果：

- 托盘操作完善
- 设置更完整
- 日常使用成本更低

## 7. 里程碑定义

## 里程碑 M1：桌宠出现

### 完成标准

- 小猫能常驻桌面
- 有透明背景
- 支持拖拽
- 支持点击
- 能记住上次位置

### 对应模块

- 桌宠外壳模块
- 基础交互模块
- 数据与本地存储模块
- UI 风格与资源模块

## 里程碑 M2：桌宠能聊

### 完成标准

- 点击小猫弹出小聊天框
- 可配置模型 API
- 能发消息和收消息
- 失败时有最基本反馈

### 对应模块

- 聊天模块
- 设置与配置模块
- 调试与可观测性模块

## 里程碑 M3：桌宠会主动

### 完成标准

- 能读取时间、电量、天气、空闲状态
- 能根据规则主动提醒
- 全屏和免打扰时默认静默

### 对应模块

- 跨平台适配模块
- 系统感知模块
- 提醒与主动行为模块

## 里程碑 M4：桌宠会记住你

### 完成标准

- 能自动提取记忆
- 能生成每日日记
- 软件内有可查看入口
- 本地文件层结构清晰

### 对应模块

- 记忆与日记模块
- 数据与本地存储模块
- 聊天模块联动

## 里程碑 M5：桌面应用完整可用

### 完成标准

- 托盘入口完整
- 设置页更完整
- 图标和基础视觉补齐
- 桌宠日常使用足够顺手

### 对应模块

- 托盘与应用控制模块
- 设置与配置模块
- UI 风格与资源模块

## 8. 建议先后顺序

推荐顺序如下：

1. 数据与本地存储模块最小版
2. UI 风格与资源模块最小版
3. 桌宠外壳模块
4. 基础交互模块
5. 设置与配置模块最小版
6. 聊天模块
7. 调试与可观测性模块最小版
8. 跨平台适配模块
9. 系统感知模块
10. 提醒与主动行为模块
11. 记忆与日记模块
12. 托盘与应用控制模块

## 9. 不建议的顺序

以下开发顺序不推荐：

- 先做记忆，再做聊天
- 先做复杂提醒策略，再做系统感知
- 先做完整设置页，再做聊天主流程
- 先做平台特化实现，再做跨平台抽象
- 先做精细视觉，再做桌宠外壳和交互

## 10. 关键阻塞点

开发中最容易卡住的几个点如下：

### 阻塞点 A：模型配置未打通

影响：

- 聊天模块无法验证
- 记忆模块无法验证

### 阻塞点 B：平台能力未抽象

影响：

- 系统感知开发会重复劳动
- Win 和 macOS 代码容易分裂

### 阻塞点 C：日志不足

影响：

- 聊天失败、提醒异常、记忆写入异常难以定位

### 阻塞点 D：数据结构不稳定

影响：

- 设置、记忆、日记、聊天记录都可能返工

## 11. 当前最合理的阶段目标

如果按自由开发推进，建议当前阶段目标定义为：

### 阶段 1

- 完成 M1
- 为 M2 做好 API 配置入口准备

### 阶段 2

- 完成 M2
- 把聊天闭环跑稳

### 阶段 3

- 完成 M3
- 让桌宠具备基础主动行为

### 阶段 4

- 完成 M4
- 让桌宠具备长期陪伴感

### 阶段 5

- 完成 M5
- 补齐桌面应用完整体验

## 12. 需要配套继续细化的文档

在本文档基础上，下一步最适合继续补的文档为：

1. 提醒策略与记忆规则文档
2. 数据结构与本地文件组织文档
3. 小聊天框与设置页交互定义文档
4. Win / macOS 平台能力适配清单

## 13. 结论

本项目后续开发不应按“功能列表堆砌”推进，而应按闭环和依赖关系推进。

推荐始终围绕这条主线：

- 先让桌宠出现
- 再让桌宠能聊
- 再让桌宠会主动
- 再让桌宠会记住你
- 最后补齐桌面应用体验

这样可以保证每个阶段都有可验证成果，也能降低返工概率。
