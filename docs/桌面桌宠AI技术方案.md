# 桌面桌宠 AI 技术方案

## 1. 项目目标

本项目目标是开发一个运行在 Windows 桌面的桌宠 AI 应用，兼顾陪伴感和轻量助手能力。产品形态不是单纯的聊天框，而是一个具备角色外观、情绪状态、主动行为、语音互动和基础系统感知能力的桌面常驻角色。

核心目标：

- 具备透明悬浮、拖拽、置顶、点击交互等桌宠基础体验
- 支持文本聊天和语音播报
- 具备基础主动性，但不高频打扰
- 具备有限的本地系统感知能力
- 支持角色状态、情绪和长期偏好记忆
- 支持后续扩展为效率助手或更强陪伴型产品

建议先以 Windows 桌面单机版为第一阶段，优先验证交互体验和角色感，再扩展更多系统集成能力。

## 2. 产品定位建议

建议采用“陪伴型 + 轻效率型”的混合路线。

原因：

- 纯陪伴型需要很强的人设、内容和持续运营能力
- 纯效率型容易与普通桌面助手同质化
- 混合型更适合个人开发者先做 MVP，既能体现桌宠差异，也能提供实际价值

建议首版定位：

- 一个有性格、会说话、会观察用户状态的桌面角色
- 可以聊天、提醒、播报、做轻量日程提示
- 不直接控制系统，不主动执行高风险操作

## 3. 总体架构

推荐采用前后端分层的本地桌面架构：

1. 表现层：负责桌宠窗口、动画、气泡、设置页、对话页
2. 行为层：负责角色状态机、情绪系统、主动发言策略、事件调度
3. AI 层：负责 LLM 对话、记忆摘要、意图识别、工具调用编排
4. 系统集成层：负责时间、天气、电量、前台应用、日历、通知等本地能力
5. 数据层：负责配置、用户画像、记忆、日志、缓存

建议逻辑拆成两个运行进程：

- `Desktop Shell`：桌面窗口和渲染层
- `Local Service`：AI 编排、系统监听、数据存储、任务调度

这样做的好处：

- UI 不会因为模型调用而卡顿
- 系统监听和任务调度更容易管理
- 后续可以把 AI 服务模块独立升级

## 4. 技术选型建议

### 4.1 桌面应用框架

首选方案：

- `Electron`
- 前端界面：`React + TypeScript + Vite`

理由：

- 上手快，生态成熟
- 对透明窗口、无边框窗口、托盘、快捷键、IPC 支持成熟
- 容易接入 Web 动画方案和 AI API
- 适合先做原型和 MVP

备选方案：

- `Tauri + Rust + React`

适合在后续对内存占用、性能、安装包体积要求更高时考虑，但首版开发成本更高。

### 4.2 角色动画

优先级建议如下：

1. 多状态 PNG 序列帧
2. `Lottie`
3. `Live2D`

建议首版使用多状态立绘或帧动画，至少实现以下状态：

- 待机
- 说话
- 开心
- 疑惑
- 生气或吐槽
- 睡眠

不建议第一版直接做 3D 模型，成本高且调试复杂。

### 4.3 AI 能力

建议拆为三类能力：

- `对话生成`：负责自然语言回复
- `行为决策`：负责是否主动发言、何时提醒、如何反应
- `记忆整理`：负责提取用户偏好、形成结构化记忆

建议模型使用方式：

- 云端 LLM：用于复杂对话、总结、意图识别
- 本地轻模型或规则：用于轻量状态判断和频繁触发逻辑

不要让所有系统事件都直接调用大模型，否则成本和延迟都不合适。

### 4.4 语音

语音输入：

- 首版可不做实时语音唤醒
- 后续接入 `Whisper` 或云端语音识别

语音输出：

- 先使用 Windows 系统 TTS 或第三方 TTS API
- 支持语速、音色、音量配置

### 4.5 本地存储

推荐：

- `SQLite`：存储用户信息、记忆、事件、配置
- `JSON`：存储简单角色配置和本地缓存

这样便于调试，也方便以后迁移。

## 5. 推荐目录结构

建议项目目录：

```text
zhuochong/
  apps/
    desktop-shell/
    local-service/
  packages/
    shared/
    prompt-engine/
    memory-core/
    event-bus/
    system-tools/
  assets/
    character/
    audio/
    icons/
  docs/
  scripts/
  data/
    dev/
```

各模块职责：

- `apps/desktop-shell`：桌宠窗口、设置页、聊天面板、托盘入口
- `apps/local-service`：状态机、调度器、AI 编排、系统监听
- `packages/shared`：类型定义、常量、通用工具
- `packages/prompt-engine`：角色设定、提示词模板、对话上下文组装
- `packages/memory-core`：短期记忆、长期记忆、摘要、检索
- `packages/event-bus`：进程间事件定义与消息桥接
- `packages/system-tools`：时间、电量、天气、前台应用等工具封装
- `assets/character`：立绘、动画资源、表情资源
- `data/dev`：本地开发数据库和调试数据

## 6. 核心模块设计

### 6.1 桌宠外壳模块

职责：

- 透明窗口渲染
- 始终置顶
- 鼠标穿透与交互切换
- 拖拽移动
- 角色点击反馈
- 右键菜单
- 托盘菜单

建议能力点：

- 支持角色在桌面边缘吸附
- 支持静音模式和免打扰模式
- 支持切换显示模式：仅角色 / 角色+气泡 / 完整面板

### 6.2 对话模块

职责：

- 接收用户文本输入
- 调用 LLM 生成回复
- 根据角色设定调整说话风格
- 支持上下文裁剪
- 生成可播报文本

建议实现：

- 将“内部推理”和“用户可见回复”分开
- 对同一轮对话输出结构化结果

结构化输出示例：

```json
{
  "visible_reply": "你今天已经连续工作很久了，要不要休息两分钟？",
  "emotion": "concern",
  "action": "show_bubble",
  "memory_candidates": [
    "用户下午容易疲劳"
  ],
  "should_speak": true
}
```

### 6.3 行为状态机模块

桌宠的关键不是回复能力，而是稳定的行为逻辑。建议做有限状态机。

推荐状态：

- `idle`：待机
- `chatting`：用户正在交互
- `observing`：监听环境变化
- `reacting`：对系统事件作出反馈
- `reminding`：执行提醒
- `sleeping`：安静模式
- `muted`：完全静音

建议内部参数：

- `favorability`：好感度
- `energy`：精力
- `talkativeness`：话痨程度
- `mood`：当前情绪
- `userFocusLevel`：用户专注程度
- `interruptCooldown`：打断冷却时间

主动行为必须由规则层先过滤，再决定是否调用模型生成话术。

### 6.4 事件调度模块

事件来源：

- 定时器
- 前台应用变化
- 电量变化
- 用户点击角色
- 聊天输入
- 系统空闲或恢复
- 日历提醒

建议事件流：

1. 事件监听器采集事件
2. 事件标准化为统一事件对象
3. 规则引擎做优先级和打断判断
4. 触发状态迁移
5. 必要时调用 AI 生成响应
6. 将结果推送给桌宠 UI

### 6.5 记忆系统模块

建议记忆分三层：

短期记忆：

- 当前会话最近 N 轮对话
- 仅用于上下文连贯

中期记忆：

- 当天事件摘要
- 当周高频互动主题

长期记忆：

- 用户偏好
- 习惯
- 禁忌项
- 角色关系进度

长期记忆必须结构化，例如：

```json
{
  "category": "preference",
  "key": "drink",
  "value": "coffee",
  "confidence": 0.82,
  "source": "conversation",
  "updated_at": "2026-04-02T10:15:00Z"
}
```

建议增加记忆治理规则：

- 低置信度记忆不写入长期库
- 敏感信息默认不保存
- 长期记忆允许用户手动删除
- 每日自动做摘要压缩

### 6.6 系统工具模块

第一阶段建议只做低风险工具：

- 当前时间
- 天气查询
- 电池状态
- 音量状态
- 当前前台应用名称
- 本地待办读取
- 日历提醒读取

第二阶段再做：

- 文件搜索
- 音乐播放控制
- 邮件摘要
- IM 消息摘要

第三阶段才考虑：

- 打开应用
- 执行脚本
- 自动操作 UI

高风险工具必须有权限开关和显式确认。

## 7. 数据流设计

典型对话流：

1. 用户点击桌宠打开聊天框
2. 用户输入消息
3. UI 通过 IPC 发给本地服务
4. 本地服务读取短期记忆和角色状态
5. 组装 Prompt 并调用 LLM
6. 对输出做结构化解析
7. 写入会话记录
8. 更新记忆候选
9. 返回 UI 展示文字和动画状态
10. 触发 TTS 播报

典型主动提醒流：

1. 调度器收到系统事件或定时事件
2. 规则引擎判断是否可打扰
3. 若可打扰，则调用行为决策逻辑
4. 生成轻量内容
5. 通知 UI 显示气泡或语音

## 8. 提示词体系设计

建议分为四类 Prompt：

- `角色设定 Prompt`
- `对话生成 Prompt`
- `行为决策 Prompt`
- `记忆提取 Prompt`

角色设定 Prompt 中应明确：

- 角色身份
- 说话风格
- 禁止事项
- 主动性边界
- 对用户的称呼策略

行为决策 Prompt 不要太开放，建议输入结构化上下文后，仅输出枚举值和简短说明，便于系统控制。

## 9. MVP 功能范围

建议首版 MVP 控制在 4 到 6 周内可完成。

MVP 包含：

- 透明桌宠窗口
- 基础角色状态动画
- 拖拽与点击交互
- 文本聊天
- TTS 播报
- 定时主动说话
- 基于时间和前台应用的简单观察
- 短期记忆
- 基础设置页

MVP 不建议包含：

- 实时语音唤醒
- 复杂 Live2D 驱动
- 自动控制电脑
- 多角色系统
- 联网代理执行复杂任务

## 10. 迭代路线

### 阶段一：原型验证

目标：

- 让角色能出现、能动、能聊、能说

交付项：

- 桌宠窗口
- 聊天面板
- 文本对话
- TTS
- 基础情绪切换

### 阶段二：行为稳定化

目标：

- 让角色“像活的”，但不烦人

交付项：

- 状态机
- 打断冷却
- 定时提醒
- 系统事件反应
- 安静模式

### 阶段三：记忆和个性化

目标：

- 让角色记住用户习惯并逐步形成关系感

交付项：

- 长期记忆
- 用户偏好管理
- 每日摘要
- 角色好感度和情绪体系

### 阶段四：扩展助手能力

目标：

- 增加轻量生产力价值

交付项：

- 日历同步
- 待办管理
- 天气、电量、工作时段提醒
- 文件检索

## 11. 性能与工程要求

建议目标：

- 常驻内存尽量控制在可接受范围
- 空闲时 CPU 占用尽量低
- 模型调用异步化，避免 UI 阻塞
- 所有系统监听都要做节流和去抖

建议工程措施：

- UI 和服务分进程
- 事件总线统一管理
- 模型调用结果缓存
- 日志分级
- 提供调试面板查看状态机和事件流

## 12. 安全与隐私

这是桌宠类产品容易被忽视但很关键的部分。

最低要求：

- 明确哪些数据会本地保存
- 默认关闭敏感数据持久化
- 所有外部 API 调用都可配置和关闭
- 所有高风险动作都需显式确认
- 提供“一键清空记忆”

建议设置权限中心：

- 天气权限
- 日历权限
- 文件访问权限
- 前台应用感知权限
- 麦克风权限
- 语音播报权限

## 13. 风险点

主要风险如下：

- 角色感弱，用户只会把它当聊天框
- 主动行为过多，造成打扰
- 模型延迟高，互动不自然
- 动画资源质量不足，影响第一印象
- 系统监听过多，造成性能压力
- 记忆设计不当，导致“记不住”或“乱记”

对应策略：

- 优先做好状态机和角色设定
- 控制主动频率，加入冷却和场景过滤
- 高频决策用规则，低频复杂回复用模型
- 先用少量高质量状态资源
- 先接低风险、低频率的系统事件

## 14. 首版开发建议

推荐开发顺序：

1. 搭好 Electron + React + TypeScript 基础工程
2. 做透明桌宠窗口和基础交互
3. 接聊天面板与模型 API
4. 加入 TTS 播报
5. 实现状态机和定时主动行为
6. 接入前台应用、电量、时间等观察能力
7. 加入短期记忆和长期偏好存储
8. 做设置页、权限页和日志面板

## 15. 推荐的首版技术栈清单

- 桌面框架：`Electron`
- 前端框架：`React`
- 语言：`TypeScript`
- 构建工具：`Vite`
- 状态管理：`Zustand` 或简单事件总线
- 本地数据库：`SQLite`
- ORM：`Drizzle ORM` 或 `Prisma`
- IPC：Electron IPC + 类型约束封装
- 日志：`pino`
- 校验：`zod`
- 任务调度：本地 scheduler 模块
- TTS：Windows TTS 或第三方接口
- 语音识别：后续接 `Whisper`

## 16. 结论

这个项目最应该优先做好的不是“大模型多强”，而是三件事：

- 桌宠外观和交互是否自然
- 行为状态机是否稳定
- 主动行为是否克制且有价值

建议你用 Electron 先快速做出 MVP，先把“像一个活的桌宠”验证出来，再逐步叠加记忆、语音和系统工具能力。这样风险最低，验证速度最快，也更适合个人开发者推进。
